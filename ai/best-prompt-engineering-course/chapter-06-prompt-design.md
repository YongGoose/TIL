# CHAPTER 06. 프롬프트를 디자인하는 방법

## 프롬프트 디자인의 세 가지 핵심 요소

- **역할(Role)**: LLM이 어떤 전문가 또는 관점에서 답변할지 설정
  - 예: "너는 10년 경력의 마케팅 전문가야"
  - 역할 설정만으로도 답변의 깊이와 톤이 크게 달라짐
- **지시(Instruction)**: 수행해야 할 태스크를 명확하고 구체적으로 기술
  - 무엇을 해야 하는지, 어떤 제약 조건이 있는지 명시
  - 모호한 표현을 피하고 구체적인 행동을 요청
- **컨텍스트(Context)**: 태스크 수행에 필요한 배경 정보와 참고 데이터 제공
  - 충분한 컨텍스트가 있어야 LLM이 정확한 답변을 생성할 수 있음
  - Chapter 05에서 다룬 벡터 서치가 컨텍스트를 가져오는 핵심 기술

## 프롬프트 디자인 프레임워크

- 체계적인 프롬프트 작성을 위한 **구조화된 접근법**
  - 매번 즉흥적으로 프롬프트를 작성하는 것이 아니라, 검증된 프레임워크를 활용
- 프레임워크의 기본 구조
  - **목표 정의**: 이 프롬프트로 달성하려는 것이 무엇인지 명확히 함
  - **입력 설계**: 어떤 데이터와 맥락을 제공할지 결정
  - **출력 설계**: 원하는 결과물의 형식과 품질 기준을 정의
  - **제약 조건 설정**: 하지 말아야 할 것, 범위, 톤 등을 명시
- 프레임워크를 사용하면 프롬프트의 **재현성과 일관성**이 높아짐

## 프롬프트 디자인 프레임워크 예제

- 실제 업무에서의 프레임워크 적용 사례
  - 블로그 글 작성, 코드 리뷰, 데이터 분석 등 다양한 태스크에 적용
- 예제를 통해 프레임워크의 각 요소가 결과물에 미치는 영향을 확인
  - 역할만 바꿔도 같은 주제에 대해 완전히 다른 관점의 답변이 생성됨
  - 출력 형식을 지정하면 후속 처리가 용이한 구조화된 결과를 얻을 수 있음

## 프롬프팅 테크닉 TOP 8

- **Zero-shot Prompting**: 예시 없이 지시만으로 태스크 수행
- **Few-shot Prompting**: 소수의 입력-출력 예시를 제공하여 패턴을 학습시킴
- **Chain of Thought (CoT)**: "단계별로 생각해봐"로 추론 과정을 유도
- **Role Prompting**: 특정 역할을 부여하여 답변의 관점과 전문성을 조절
- **Instruction Prompting**: 명확하고 구체적인 지시사항을 제공
- **Self-Consistency**: 같은 질문에 여러 번 답변을 생성한 뒤 가장 일관된 답변을 선택
- **Generated Knowledge Prompting**: LLM에게 먼저 관련 지식을 생성하게 한 후, 그 지식을 활용해 최종 답변을 생성
- **ReAct (Reasoning + Acting)**: 추론과 행동을 번갈아 수행하며 외부 도구를 활용하여 문제를 해결

## 프롬프팅 확장 테크닉: 예시를 제공하고 생각하게 만들기

- **Few-shot + CoT 결합**: 단순히 예시를 주는 것에서 더 나아가, 추론 과정이 포함된 예시를 제공
  - 입력 → 추론 과정 → 출력 형태의 예시를 보여주면 LLM도 동일한 추론 패턴을 따름
- **Zero-shot CoT**: 예시 없이 "Let's think step by step"만 추가해도 추론 능력이 향상됨
  - 복잡한 수학 문제, 논리 추론, 다단계 분석에서 특히 효과적
- 핵심: LLM이 **생각하는 과정을 명시적으로 거치도록** 유도하는 것이 정확도 향상의 핵심

## 프롬프팅 확장 테크닉: 전략을 짜고 스스로 평가하기

- **Plan-and-Solve Prompting**: 문제를 풀기 전에 먼저 계획을 세우도록 유도
  - "먼저 이 문제를 해결하기 위한 계획을 세워줘" → 계획에 따라 실행
- **Self-Evaluation / Self-Critique**: LLM에게 자신의 답변을 스스로 검토하고 개선하도록 요청
  - "네 답변을 다시 검토하고, 잘못된 부분이 있으면 수정해줘"
  - 자체 평가를 통해 환각이나 오류를 줄일 수 있음
- **Reflexion**: 이전 시도의 결과를 반성하고 개선된 답변을 생성하는 패턴
- 이러한 테크닉들은 LLM의 **자기 교정 능력**을 활용하여 답변 품질을 높이는 전략

## 포맷팅

- 프롬프트의 **구조와 형식**이 LLM의 이해도와 출력 품질에 직접적인 영향을 미침
- 효과적인 포맷팅 전략
  - **구분자(Delimiter) 활용**: ```, ---, """, XML 태그 등으로 섹션을 명확히 분리
  - **마크다운 활용**: 제목, 리스트, 표 등으로 프롬프트를 구조화
  - **번호 매기기**: 여러 지시사항이 있을 때 순서를 명시
- 출력 포맷 지정
  - JSON, CSV, 표, 마크다운 등 원하는 출력 형식을 명시적으로 요청
  - 예시 출력을 함께 제공하면 형식 준수율이 높아짐
- 잘 구조화된 프롬프트 = **LLM이 의도를 정확히 파악**할 수 있는 프롬프트

## 프롬프트 체이닝

- **프롬프트 체이닝(Prompt Chaining)**: 복잡한 태스크를 여러 단계의 프롬프트로 분리하여 순차적으로 실행하는 기법
  - 이전 단계의 출력이 다음 단계의 입력이 됨
- 체이닝이 필요한 이유
  - 하나의 프롬프트로 복잡한 태스크를 처리하면 품질이 떨어짐
  - 단계별로 분리하면 각 단계의 **정확도와 제어력**이 높아짐
- 체이닝 예시
  - 1단계: 긴 문서를 요약 → 2단계: 요약을 기반으로 핵심 인사이트 추출 → 3단계: 인사이트를 기반으로 보고서 작성
- 프롬프트 체이닝은 **AI 에이전트**와 **워크플로우 자동화**의 기반이 되는 핵심 개념

## 좋은 프롬프트 만들기

- 좋은 프롬프트의 핵심 원칙
  - **명확성**: 모호한 표현을 피하고 구체적으로 작성
  - **구체성**: "잘 써줘"보다 "3문장으로 요약하고, 핵심 키워드 5개를 추출해줘"
  - **맥락 제공**: LLM이 모르는 정보는 프롬프트에 직접 포함
  - **제약 조건 명시**: 하지 말아야 할 것, 출력 길이, 톤 등을 지정
- 반복적 개선(Iteration)이 핵심
  - 한 번에 완벽한 프롬프트를 만들기 어려움
  - 결과를 확인하고 프롬프트를 조금씩 수정해가며 최적화
- 자주 하는 실수
  - 한 프롬프트에 너무 많은 태스크를 넣는 것 → 태스크를 분리할 것
  - 원하지 않는 것만 나열하는 것 → 원하는 것을 직접 명시할 것
  - 컨텍스트 없이 질문하는 것 → 충분한 배경 정보를 제공할 것
- **프롬프트 엔지니어링은 기술이자 예술**: 체계적 방법론을 따르되, 창의적 실험을 통해 최적의 결과를 찾아가는 과정
